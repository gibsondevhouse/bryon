# Bryon v1.5 Kickoff Plan

## Summary

V1.5 is the Projects release. It builds on rich-chat v1 by adding project-scoped organization, prompt defaults, reusable prompt presets, richer memory editing, and reusable project files. It is not an agent/tool-loop release.

## Local Hard Gate

Run this before and after each phase:

```sh
pnpm gate
```

If `pnpm gate` fails, stop phase progression and fix the regression before continuing.

## v1.5 Phase Order

1. Projects schema and API: project CRUD, optional `chats.project_id`, archive state, non-destructive migrations.
2. Project UI shell: project list, create/rename/archive, move chats into/out of projects.
3. Project files: upload reusable project files, extract text, select files for a chat turn explicitly.
4. Prompt library: edit default Bryon prompt, create presets, apply presets, set project prompt overrides.
5. Memory editing: global/project scopes, enable/disable, archive-ready structure, origin metadata if auto-memory is later added.
6. Project knowledge search: search extracted project-file text; add embeddings/RAG only after explicit project-file selection works.
7. Hardening: migration compatibility, failure states, upload/search limits, visual and e2e coverage.

## Immediate Next Phase: Projects Schema and API

### Goal

Add project containers without changing global chat behavior.

### Deliverables

- `projects` table with name, description, prompt override, memory fields, archive/timestamps.
- Nullable `chats.project_id` with legacy chats remaining global.
- Project CRUD endpoints.
- Chat create/update support for assigning or clearing a project.
- Tests for migration compatibility and global-chat fallback.

### Exit Criteria

- Existing chats still load with no project.
- New projects can be created, renamed, archived, and listed.
- Chats can be assigned to a project and moved back to global.
- All gates pass.

## Execution Rules

- Do not reintroduce public persona switching.
- Do not add model-controlled tool loops.
- Keep uploads and web lookup explicit user controls.
- Preserve old DB compatibility; never destructively drop legacy tables/columns as part of the scope reset.
