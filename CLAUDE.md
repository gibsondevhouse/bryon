# Project Configuration

- **Language**: TypeScript
- **Package Manager**: pnpm
- **Add-ons**: none

---

## Bryon

Local AI chat client. Full-stack SvelteKit app that talks to a locally-running Ollama instance. Single-user, single-machine. No cloud, no accounts.

---

## Prerequisites

Ollama must be running before the app will work:

```sh
ollama serve
ollama pull gemma4:e4b
```

## Key constraints

- **Browser never talks to Ollama directly.** All LLM calls go through `POST /api/chats/:id/stream`.
- **`lib/server/` is server-only.** Never import from the client side. Vite will error if you try.
- **DB opens once at boot** in `hooks.server.ts`. Never create new connections in endpoints — use the singleton from `db/client.ts`.
- **All API responses are JSON** except `/api/chats/:id/stream` (SSE / `text/event-stream`).
- **Messages are immutable** after creation. No edit or delete on individual messages.
- **Personas are live-injected** — fetched fresh on every turn, not snapshotted per chat.

## Release-Ready Quality Standards

Every change — backend or frontend — must ship at production quality. Bryon is a finished product, not a prototype. Code that violates these standards must not be merged. Treat each PR as if it were the final shipping cut.

### Backend (server / API / DB)

- **Validate at boundaries.** Every endpoint validates its request body with a Zod schema from `src/lib/shared/schemas.ts`. Return `400` with a structured error on validation failure.
- **Typed errors only.** Throw structured errors (`{ code, message, details? }`); never leak stack traces or raw SQL/Ollama errors to clients. Pino logs the full error server-side.
- **No partial writes.** Multi-step DB operations run inside `db.transaction(() => { … })`. If any step fails, the whole operation rolls back.
- **Accurate status codes.** `200` only when the requested action completed. `201` for create, `204` for delete, `404` for missing entities, `409` for conflicts, `422` for invalid lifecycle transitions, `503` when Ollama is unreachable.
- **No silent failures.** Every `try/catch` either re-throws, returns a typed error response, or logs at `error` level with structured context. Never swallow.
- **Streaming endpoints clean up.** Aborted SSE streams must close the underlying Ollama request and persist the partial message with `ms_total = null`.
- **Migrations are reversible in intent.** Never drop a column or table without a deprecation step. Schema changes ship with a Drizzle migration generated by `pnpm db:generate`; never hand-edit migration SQL.
- **No `any`, no `// @ts-ignore`, no unjustified `// biome-ignore`.** If the type system complains, fix the type.
- **Tests required for new logic.** Any new service function, lifecycle transition, or routing decision needs a Vitest unit test. Any new user-facing flow needs a Playwright e2e test.
- **Logs are structured.** Pino fields only — no string interpolation in log messages. Include `chatId`, `planId`, and similar identifiers where applicable.
- **Performance budgets.** List endpoints ≤ 100ms p50, ≤ 300ms p95 against a populated DB. Stream first-token ≤ 1000ms when Ollama is warm.

### Frontend (Svelte / UI / UX)

Polish is not optional. The product should feel finished, not scaffolded.

- **Every state is designed.** Loading, empty, error, and success states all have intentional UI — never a blank page or a raw error string.
- **Empty states name the next action.** "Create your first plan" — not "No plans found".
- **No raw fetch errors in the UI.** Surface errors via the toast/notification system with human-readable copy and a retry affordance where appropriate.
- **Skeleton or spinner on every async boundary.** No layout shift when data loads. Reserve space with skeleton components matching final content shape.
- **Optimistic updates where safe.** Create/update/delete actions update the UI immediately and reconcile against the server response. Rollback on failure with a toast.
- **Forms are accessible.** Every input has a `<label>`, every required field is marked, error messages are tied to inputs via `aria-describedby`. Keyboard navigation works end-to-end (Tab order, Enter to submit, Esc to cancel).
- **Keyboard shortcuts documented.** Any shortcut shows up in a `?` help overlay. Don't add hidden shortcuts.
- **Responsive down to 1024×720.** The app targets desktop but must not break on smaller laptop screens. Sidebar collapses gracefully.
- **Motion is purposeful.** Use Svelte transitions for state changes. Respect `prefers-reduced-motion`. No gratuitous animation.
- **Visual consistency.** Reuse components from `src/lib/components/` and primitives from `src/lib/ui/`. Don't ship one-off button or input styles. Spacing, typography, and color come from the existing token system in `src/app.css`.
- **Copy is final.** No placeholder text ("Lorem ipsum", "TODO", "foo"). Sentence case in body copy; title case only for top-level page titles. Headings, button labels, and error text are written for users — clear, concise, action-oriented.
- **Icons via the established icon set.** Don't import a new icon library for one icon.
- **Dark mode parity.** Every new component renders correctly in both themes. Test both before merging.
- **No dead routes.** A route in the sidebar must be functional. Stubs are acceptable only when clearly labeled "Coming in Phase NNN" with a visible link to the roadmap — never silently empty.
- **Streaming UI is polished.** Token streaming renders smoothly (no character jitter), shows a clear streaming indicator, and offers a Stop button that actually aborts the underlying request.
- **No console output in shipped code.** No `console.log`, no `console.warn`. Use the server logger for diagnostics.
- **Destructive actions confirm.** Delete, archive, or anything that loses data requires explicit confirmation, and the confirmation copy names what will be lost.

### Pre-merge checklist (run before declaring work done)

1. `pnpm lint` — passes with zero warnings
2. `pnpm format` — already applied
3. `pnpm test` — all unit tests green
4. `pnpm test:e2e` — affected e2e suites green (full suite for shared flows)
5. `pnpm build` — production build succeeds
6. Manual walkthrough of every state (loading / empty / error / success) for any changed UI
7. Both light and dark themes verified for any changed UI
8. Keyboard-only walkthrough of any changed form or interactive surface
9. No `console.log`, no `TODO`, no commented-out code in the diff
10. Migration drift check passes (`pnpm db:migrate` against a fresh DB applies cleanly)

## SSE streaming format

The `/api/chats/:id/stream` endpoint emits `text/event-stream`. Event types:

```text
event: token
data: {"delta":"Hel"}

event: meta
data: {"msToFirst":412,"tokensIn":287}

event: error
data: {"code":"MODEL_NOT_FOUND","model":"gemma4:e4b","message":"..."}

event: done
data: {"id":"01J...","tokensOut":58,"msTotal":1830}
```

A `ms_total = null` on a persisted assistant message means the stream was interrupted or cancelled.
